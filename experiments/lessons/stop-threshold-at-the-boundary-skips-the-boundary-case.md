# [2026-07-21] An early-stop threshold set AT the reporting boundary silently skips the case that starts there [TRAP]

`mu_ladder.py` defaults to `--stop-mu 13` and breaks out of the rung loop as soon as `best_mu <= stop_mu`. `aca_115` (AK(3)'s class) *starts* at μ = 13, so in the 124-class rungs-20/beam-32 run it stopped after rung 1 and wrote `n_orbits_seen: 2`. Read against the other 123 classes (~1,300–1,950 orbits each) that row looks like a spectacular isolation result — AK(3) inert under the whole CoV family. It is not a result at all: the class was never laddered. The stop threshold is set one above the actual solve criterion (μ ≤ 12, `results/stable_ac/theory/MU_CRITERION.md`) so that a μ=13 landing gets flagged for a human, which is right — but it also means any class already sitting on the flag value gets zero exploration, and the row it writes is indistinguishable in shape from a class that was fully explored and found nothing.

Rerunning that one class with `--stop-mu 12` (`--data data/ms_unsolved_reps/ak3_only.csv`, ~100 s single-threaded locally) gives the real measurement: **1,508 orbits reached, median fan-out for the set, and nothing below μ = 13** — of those, exactly 2 sit at 13, 5 at 14, 35 at 15, and the mass is at 17–25. AK(3) is not inert in reachability; it is inert in *descent*. The second μ=13 orbit is `YYXXyx|YYYxyXX`, the already-documented orbit-2 of AK(3)'s length-13 AC class (`experiments/stable_ac/cov/ak_3_universal_test/RESULTS.md`), rediscovered here in 3 CoV hops (z = `xy`, `yX`, `Xy`) — a consistency check on the machinery, not new information.

**Rule:** when a run's early-stop threshold and its inputs live on the same scale, check whether any input already satisfies the threshold — those rows are *unexplored*, not *explored and negative*, and nothing in the row's schema says so. Before quoting a low work-count (`n_orbits_seen`, pops, nodes) as evidence of a property of the object, confirm the run was not cut short by its own stopping rule; a distribution outlier at the low end is a stopping artifact until proven otherwise. Prefer either a threshold strictly below every input value, or a recorded `stopped_early` reason in the row.
