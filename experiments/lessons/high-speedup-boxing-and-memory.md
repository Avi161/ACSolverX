# [2026-07-08] HIGH_SPEEDUP heavy solver: boxing was the cost, memory was the cap [WORKS]
Heavy runs (261 unsolved reps, 1M nodes, mrl=48) were infeasible: measured **43.4× discovery ratio** (30k pops → 1,302,816 states), **554 B/state** → ~43M states / **~25 GB per presentation**, ~14 min each ⇒ ~60 h serial. `cProfile` showed the time was **numba boxing**, not math: 3.5M `_numba_unpickle` calls per 8k pops, because `get_neighbors_with_moves_nj` returns a Python list of ~440 tuples-of-ndarrays per node. And 50 GB fits exactly ONE presentation, so cores sat idle — **memory reduction is what unlocks parallelism, not a separate fix.**
**Solution** (`HIGH_SPEEDUP` config bool, default False; normal path bit-for-bit unchanged): (1) drop `new_seen` — it only fed the final `min()`/`max()` scan → track min/max incrementally; (2) `visited` becomes a `set`, `move_in` gone (no path) → solved presentations are re-solved by the normal solver **after the pool tears down** (it needs full RAM) and get `path_recovered: true`; (3) packed `bytes` keys; (4) `expand_node_nj` does neighbours+reduce+canonicalise inside numba returning one fixed-width `uint8` buffer (use `np.empty`, not `np.zeros` — only written cells are read), plus `multiprocessing` over presentations with the **parent as sole jsonl writer** (no lock, resume untouched).
**The key insight that made (3) free:** the packed key must SORT IDENTICALLY to `(str, str)`, because the heap tie-break is `(priority, depth, key)` and `depth` is fixed by first-visit order. A 1-byte order-preserving alphabet (`X→1 < Y→2 < x→3 < y→4`, matching ASCII) with a `0x00` separator (below every symbol, reproducing str's shorter-prefix-is-smaller) does exactly that — verified on 3,681 pairs. Also: `expand_node_nj` must enumerate in the same `target→jsign→k1→k2` order, else `depth` shifts and pop order changes.
**Verified:** 400 runs (200 pres × 2 cyclic modes) — 0 mismatches on `solved`/`nodes_explored`/`min_relator_length`/`max_relator_length`/`max_relator_length_expanded`; `min_relator` and `max_relator_expanded` strings identical too. Only `max_relator` differs (353 cases) — all **equal-total-length ties**, because `max(set, key=…)` picks arbitrarily among ties while the incremental scan picks the first. Parallel == serial; recovered paths replay to trivial. **Measured: 554→213 B/state (2.6×), 678→281 µs/node (2.41×), identical discovered counts (1,302,816).** ⇒ ~9.2 GB & ~4.7 min per presentation @1M ⇒ 4 workers on 50 GB ⇒ ~5 h instead of ~60 h.
**Rules:** `HIGH_SPEEDUP` must NOT enter `_run_prefix` (it doesn't change results — like the `use_*` toggles). It does **not** make the 261 solvable; it only completes the experiment.
**[2026-07-09 CORRECTION] "wasteful on easy data" overstated it.** Measured on ms640 idx 620-639 @30k, mrl=24 (10 solved / 10 budget-capped): normal 210.3s, heavy@4workers 80.8s (**2.60x**), heavy@1worker 131.6s (**1.60x**) — parity on solved/nodes_explored/path_length. Heavy is faster even there. The right model is `heavy ≈ all_work/2.4/W + work_in_SOLVED_presentations` vs `normal ≈ all_work`: the heavy pass is cheap and parallel, but the path-recovery re-solve is FULL PRICE and **serial in the parent** (no pool). So the win scales with the share of total node-cost spent on presentations that never solve — huge for the 261 unsolved reps (recovery ≈ 0), eroding on ms640 where ~95% solve, and it can invert at W=1. Also: `GB_PER_PRES=9.0` is calibrated for 1M/mrl48; at 50k/mrl24 a search uses a few hundred MB, so leaving it at 9.0 silently under-provisions workers (auto gives 4 where ~8 would fit) and you pay the double-solve for little benefit.
