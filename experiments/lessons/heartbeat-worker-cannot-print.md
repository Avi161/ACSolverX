# [2026-07-09] Live nodes/s heartbeat — a pool worker cannot print into a notebook [TRAP]
`PROGRESS_EVERY` only prints when a presentation *finishes*; at a 1M budget that is minutes of silence, so there was no way to see CPU throughput mid-solve. Added `HEARTBEAT_EVERY_S` (0 = off): both solvers take an optional `progress` callback, invoked every `_HB_CHECK_EVERY = 1024` pops; the callback rate-limits itself by wall clock. Overhead unmeasurable (146 vs 152 µs/node, heartbeat-on measured *faster* — pure noise). **The trap:** in the parallel (`HIGH_SPEEDUP` + `n_workers > 1`) path a worker's `print()` is silently DROPPED in Jupyter/Colab — ipykernel's stdout proxy forwards through a background `pub_thread` that a forked child does not inherit. Workers therefore `put_nowait` samples onto an `mp.Queue` (passed via `Pool(initializer=...)`, which survives both `fork` and `spawn`) and a daemon thread in the PARENT drains and prints. **Second trap:** dropping a finished presentation by a `2*every` staleness window is wrong — its last rate keeps inflating the `agg nodes/s` and the "N solving" count for two more heartbeats (observed live). `_solve_one` now sends a `(pres_id, None)` done-sentinel in a `finally`; the staleness window survives only as a backstop for a worker that dies. **Rules:** verify the parallel path by FORCING `N_WORKERS=2` (on 16 GB `_auto_workers` returns 1, so `n_workers > 1` never fires and the whole Queue/monitor path goes untested); test under both start methods (macOS defaults to `spawn`, Colab to `fork`); `HEARTBEAT_EVERY_S` is result-neutral so it must stay OUT of `_run_prefix`. Verified `progress=None` vs a live callback returns bit-identical stats dicts on 4 presentations × both solvers.

**Simpler alternative when workers already append per-worker jsonls** (the cov chunked runs): skip the Queue plumbing — the PARENT scans the workers' files between `join(timeout)` calls and prints the aggregate (`run_cov._ChunkFilesHeartbeat`, 2026-07-20). The scan must skip an unparseable in-flight tail line and must NEVER repair/truncate a file another process owns.
