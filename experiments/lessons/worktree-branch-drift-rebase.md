**Rules:** on a long task, re-check `git log` on the target branch before pushing — "my tests pass" means nothing against a stale base. Check `git worktree list` **before** starting: a sibling worktree means a concurrent session, and its branch may already fix part of your task — expect duplicate fixes and a textual conflict, resolve by keeping either. Never edit another session's worktree. When a rebase lands you on top of someone else's change to the same file, **re-run both test tiers**: two independently-green changes to one file say nothing about their merge. When a neighbouring module already has tests, test the *seam*, not the internals; two suites asserting the same behaviour is two places to update. And a new root `pytest.ini` with `testpaths` silently changes what a bare `pytest` collects — enumerate every existing test directory.
