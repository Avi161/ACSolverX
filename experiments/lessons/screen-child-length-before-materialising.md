# [2026-07-21] Price a child by its reduced length before building it — the cap prune is the hot loop [WORKS]

NB6's tier-300 climbs ran at ~1 pop/s on Colab (~3.8 pops/s locally) even after the fused-kernel rewrite, and the fused rewrite was not the fix it looked like. `expand_node_nj` concatenates, reduces BOTH relators, and only then tests `len(a) > cap or len(b) > cap`. A climb saturates at the cap by construction — at cap 300 the popped state is ~300+300, so essentially every one of the ~90k seam-valid children reduces to ~600, fails that test, and is discarded after O(L) work it never needed. The fused kernel had removed the Python↔numba boundary crossings, which is ~2× on descends (short states, most children survive) and ~nothing here: the boundary was never the cost at saturation, the discarded materialisation was.

The reduced length of `rot_i ++ rot_o` does not require building it. Both halves are freely reduced (rotations of a cyclically reduced relator are), so the only free cancellation is the seam cascade: length = `n1 + n2 - 2c` with `c` the number of cancelling seam pairs, and the cyclic pass then indexes the virtual word `rot_i[:n1-c] ++ rot_o[c:]` — all O(cancelled), no allocation. `expand_node_topk_nj` does that as pass 1, then concatenates/reduces/canonicalises/encodes only the `topk` survivors the caller would have kept. It also drops the old kernel's 217 MB-per-pop `(4·(n1+1)·(n2+1), 2·cap)` scratch buffer for int64 length arrays (~7 MB), and reduces the untouched relator once per target instead of once per child.

Measured on aca_0, cap 300, identical results: one expand at the saturated state (256+299) **396 ms → 14 ms (29×)**; a 60-pop climb **15.6 s → 0.7 s = 3.83 → 82.6 pops/s (21.5×)**. At cap 150 the saturated expand is 66 ms → 4 ms.

This is only safe because it is bit-identical, and that is what makes already-written jsonl rows still resume-compatible: selection is a stable sort on the same reduced total, so `expand_node_topk_nj(..., sgn, topk)` reproduces `expand_node_nj` + the caller's `np.argsort(sgn * totals, kind='stable')[:child_cap]` child-for-child, and generation order when the survivor count is ≤ `topk`. `test_fused_expansion_matches_the_reference_child_path` pins the whole chain (reference expansion → fused → screened, both directions, `topk ∈ {0, 512, 5}`, fat state, cap-prune and truncation both asserted to have fired), and `test_seam_length_screen_equals_a_full_reduce` pins the screen against `reduce_relator_nj` on 400 random reduced pairs. The precondition — both halves freely reduced — holds for every state a search produces (children are reduced then canonicalised); it would NOT hold for an arbitrary unreduced input.

**Rule:** when a kernel enumerates children under a cap, compute the cheap quantity the cap tests (here: reduced length, from the seam alone) and prune *before* materialising. Profile which half of a fused kernel is actually hot before assuming the fusion was the win — at saturation the answer is the work you throw away, not the boundary you crossed. Any such rewrite must be proven child-for-child identical, or previously written rows stop being resumable.
