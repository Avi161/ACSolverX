# [2026-07-09] Heartbeat silent for 2x the interval — tested at 2s, shipped at 90s [TRAP]
User set `HEARTBEAT_EVERY_S=90` on the real 261x1M run and saw NOTHING after 90s. Two compounding phase bugs, both invisible at the `every=2` I tested with: (1) `_Heartbeat` waited a **full** `every` before its FIRST sample, and the worker's clock starts before numba JIT eats the first seconds; (2) `_hb_monitor` initialised `last_print = time.time()` and then ran `last_print = now` **even when it printed nothing** — so the T+90s tick found `live` empty (the worker's first sample lands moments later), burned the whole period, and the first block appeared at T+180s. Whether it was 90s or 180s was a pure race on which arrived first. **Fixes:** `last_print = 0.0` (first arriving sample prints at once); `continue` WITHOUT advancing `last_print` when `live` is empty; `_Heartbeat` emits its first sample after `min(first_after=10s, interval)`; and pool workers sample at `hb_s/2` (the parent prints every `hb_s`, so sampling at `hb_s` would build `agg` from rates up to a full period stale). Verified with a timestamped wrapper: first block at 8.5–11s, then exact cadence, across serial / fork / spawn. **Rules:** a rate-limiter's *first* emission is a separate design decision from its steady-state period — long silence at startup is indistinguishable from a hang. Never advance a "last fired" timestamp on a tick that produced no output. And **test periodic code at the production interval** (or with a timestamped log): a 2s test hides every phase/race bug that a 90s interval exposes.
