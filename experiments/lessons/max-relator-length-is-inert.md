# [2026-07-09] MAX_RELATOR_LENGTH is nearly INERT under best-first search [WORKS]
Measured, not assumed. Both datasets top out at a **17-symbol** single relator (`ms640_solved` and `ms_reps_unsolved`, file slot 24), so a 24 cap is representable. Because the heap is ordered by TOTAL length, the frontier stays short and the per-relator cap almost never binds: on `ms_reps_unsolved` pres 0-2, cap24 vs cap48 give **field-for-field IDENTICAL** stats dicts at 50k AND 200k nodes (longest single relator seen: 19-23). The first divergence is pres 2 @200k (longest relator hits 24) — and even then only `max_relator`/`max_relator_length` differ, NOT `solved` or `nodes_explored`: the pruned long-relator states are discovered (stored) but never popped within budget.
**Cost of the cap at depth** (pres 2, 300k nodes, heavy): cap24 = 61.1s, 14,646,796 discovered, 2.69 GB peak; cap48 = 73.2s, 19,465,394 discovered, 2.86 GB peak ⇒ cap24 is **1.20× faster, 25% fewer discovered states, ~6% less peak RSS**. Note `expand_node_nj` costs O(len_i × len_o) on ACTUAL lengths, not the cap (only the `np.empty((ub, 2*cap))` column count scales) — which is why the caps cost the same until the cap actually binds.
**Solve parity across caps:** ms640 idx 621 and 630 solve at cap48 with EXACTLY the cap24 `nodes_explored` (15,371 / 12,429) and `path_length` (500 / 79); heavy == normal at both caps; recovered paths replay to trivial. Caveat: on those the cap never bound (relators ≪ 24), so this proves cap48 is harmless, not that the >24 pruning branch is exercised there.
**Determinism:** no RNG, fixed enumeration order in `expand_node_nj`, total-order heap on `(priority, depth, key)` ⇒ `solved`/`nodes_explored`/`path_length`/min+max **lengths** are platform-independent (local result carries to Colab). Exception: the NORMAL solver's `min()`/`max()` over the `new_seen` **set** breaks ties by set iteration order (PYTHONHASHSEED), so `min_relator`/`max_relator` **strings** can vary per process. Lengths never do.
**Rules:** don't lower the cap to buy speed — it strictly shrinks the search space and can only reduce the solve rate (substitution moves work by temporarily LENGTHENING a relator, per Two-Hump). To bound a run, cut `SUBSET` or `BUDGET`, which shrink the experiment openly. `MAX_RELATOR_LENGTH` is in `_run_prefix`, so changing it correctly starts a new jsonl + W&B run.
