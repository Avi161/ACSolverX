# [2026-07-21] CoV chains do NOT concatenate across canonicalization — the junction is an automorphism [TRAP]

The μ-ladder records descent chains as z-word lists. A second-generation ladder run started FROM the floor bench's rows — which are `aut_canon` canonical representatives, not the concrete pairs the first chain reaches. Naively concatenating the two z-lists and replaying from the original class rep FAILED verification (`composed chain fails for aca_100_floor`): the CoV transform is representative-sensitive (`subword_candidates` reads the concrete strings), so the second chain's z-words need not even be subwords at the pair the first chain lands on. Worse, the failure was masked into commit `e89e284`, whose message claimed "composed chains verified" while the heredoc had died before writing — the newline-separated `git` commands ran anyway (a cousin of the zsh `&&`-chain lesson: verification and commit must be coupled by `&&`, never by mere adjacency). Corrected in `63b9849`.

The correct semantics: `source ~st (chain₁) ~st P₁ ~st (Thm-3 automorphism to the canonical rep R) ~st (chain₂ from R) ~st target`. The junction automorphism is a legal stable move (PROOFS.tex Thm 3), so the composite is still a stable certificate — but it is a THREE-segment certificate, and verification must replay chain₁ to P₁, check `aut_canon(P₁) = R`, then replay chain₂ from R. The bench encodes the junction as `|aut|` inside the `chain` column.

**Rule:** never concatenate CoV z-chains recorded against different representatives — verify each segment from the representative it was actually derived from, with the canonicalization junction explicit; and couple any verification script to its commit with `&&` so a failed check can never ship a claiming message.
