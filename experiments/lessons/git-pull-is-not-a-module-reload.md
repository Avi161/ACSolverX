# [2026-07-09] `git pull` is NOT a module reload — stale sys.modules ate a shipped fix [TRAP]
User pulled a heartbeat fix, re-ran SETUP + RUN, and still saw the old (broken) behaviour. Cause: SETUP's `git reset --hard` updated `experiments/run_baseline.py` on disk, but `experiments.run_baseline` was already in `sys.modules` from the previous RUN, so the RUN cell's `from experiments.run_baseline import run_dataset` returned the **stale** module. Proven locally: import, rewrite file, re-import → still the old value; after `del sys.modules[...]` + `importlib.invalidate_caches()` → new value. Interrupting a cell does NOT clear `sys.modules`; only a runtime restart does. Worse, the diagnostic line (`heartbeat: nodes/s every 90s`) existed in BOTH commits, so the output looked like the new code was running. **Fix:** SETUP now purges `experiments*` from `sys.modules` + `importlib.invalidate_caches()` right after `os.chdir(REPO_ROOT)`. **Rules:** after pushing a `.py` fix, tell the user to **Runtime → Restart session** (or re-run the patched SETUP), never just "re-run the cell"; when diagnosing "my fix didn't take", first prove WHICH code is loaded (`inspect.signature`/`inspect.getsource` of a symbol that exists only in the new commit) rather than trusting a log line that both versions emit. Beware `.pyc` staleness too: a rewrite with identical size within the same mtime second reuses the cached bytecode (hit while writing the repro).
