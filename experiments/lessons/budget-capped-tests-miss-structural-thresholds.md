# [2026-07-24] A budget-capped test can leave whole code paths structurally unreachable — force each threshold below the ceiling [WORKS]

The repo's 1,000-node test ceiling rests on a true theorem: a search at budget `B` is exactly the first `B` pops of any longer search, so a bigger budget buys a slower repro, never different *search behaviour*. What the `hcompact` review (independent adversarial pass, 2026-07-24) made precise is that the theorem covers **semantics only**. A solver's *infrastructure* has its own numeric thresholds, and if a threshold sits above what the test ceiling can reach, the path behind it ships completely unexercised — a green suite says nothing about it:

- **The chunk boundary.** `_HB_CHECK_EVERY = 1024`, so every budget ≤ 1,000 search runs as a SINGLE `_run_chunk` call. The njit↔Python re-entry — stats round-tripping through the `st[]` array, per-state arrays surviving the boundary — was never crossed by any test in the suite, at any budget the rules allow.
- **The grow path.** `est_states(1000) × 1.5` reserves ~119k slots against ~73k discovered, so reservation-exceeded → grow-copy-rehash never fires either. For `hcompact` that path must carry two extra arrays (`score`, `seg`); dropping them would reorder every state still in the heap, and no budget-legal test would have noticed.

Both were closed without touching the ceiling, which is the point: the fix is to **move the threshold below the test, never the test above the ceiling**. `test_chunk_boundary_preserves_the_search` monkeypatches `_HB_CHECK_EVERY` to 100 so a 500-pop burn crosses the boundary four times; `test_growth_preserves_the_search` passes `reserve_states=1` so the arena floors at ~10.6k slots and doubles several times mid-burn. Each then requires full-dict equality with the reference solver, first-seen strings included. (Same family as the config-parameter version of this trap: `cap-monotonicity-vacuous-guard.md` — a guard that never fires makes a green test meaningless.)

**Rule:** when a test suite runs under a size ceiling, enumerate the implementation's *structural thresholds* — chunk sizes, reservation floors, resize triggers, buffer high-water marks — and check each against the largest test that can run. Any threshold the ceiling cannot reach gets a dedicated test that drags the threshold down (monkeypatch the constant, shrink the reservation) rather than pushing the workload up. "The first B pops are identical" licenses capping the *search*; it says nothing about the machinery underneath it.
