# Data Distribution Plan — d-o-t Regressor

*Working draft (under review). Precedes model planning (PLAN.md §4+); refines PLAN.md §2/§5/§6.*
*Scope: build the right TRAINING DISTRIBUTION — the balance of solved vs genuinely-hard /
unsolved (AK(3)-class) presentations — before any model code. ("regressor" → CLARIFYING #1.)*

---

## 0. The problem this plan exists to solve

The model must predict d-o-t well **on hard presentations**, where relator length carries no
signal. The raw data fights that:

**Finding 1 — the two humps overlap at every length** (verified on `dot_archive.jsonl`):


| total_len | #solved | solved median d-o-t | #censored (hard) | censored frac |
| --------- | ------- | ------------------- | ---------------- | ------------- |
| 13        | 880     | 6                   | 7                | 0.8%          |
| 14        | 972     | 7                   | 20               | 2%            |
| 15        | 3,086   | 8                   | 266              | 8%            |
| 16        | 3,856   | 8                   | 659              | 15%           |
| 17        | 8,935   | 9                   | 4,001            | **31%**       |


Length predicts *probability of being hard* (0.8%→31%) but **not** the d-o-t of the solvable
ones (median ~6→9). Mostly-easy with a same-sized hard minority at every length — so the model
must separate them by **structure**, not size.

**Finding 2 — the look-alike trap.** `(YXyxYx, x³y⁻⁴)` is solved (d-o-t 12) in the data, yet
canonical **AK(3) =** `(YXYxyx, x³y⁻⁴)` (same power relator) is the open counterexample and is absent so the model would rank AK(3) as easy, unless we have found a model that can clearly understand the difficulty distribution, which is very hard.

**Finding 3 — short-hard examples barely exist.** Only **8** censored rows are length ≤13; the
other 4,946 are length 14–17 (81% at 17). Raw data ≈ "short ⇒ easy."

---

## 1. The five distribution principles

1. **Balance by hardness, not count** — reweight to a target mixture (§2); don't resample.
2. **Decouple length from hardness** — add short-hard anchors, no length feature, length-stratified val metric.
3. **Populate the valley** (d-o-t 11–50) — keep beam's detour states; (Phase 2) add automorphic hard-but-solvable images.
4. **Teach hardness honestly** — unsolved rows get a one-sided hinge (penalize only *under*-prediction); no fake labels.
5. **Hold out the goal** — a frozen `counterexample_eval` (AK-family, never trained) tests hardness recognition. For this to be meaningful, training must still contain *other* hard/unsolved examples and a good solved↔unsolved balance — which §2 enforces.

---

## 2. Target distribution (the solved ↔ unsolved balance)

Effective **loss mass** target (by weights, not duplication):


| region                  | def (d-o-t)        | current | target  | lever                    |
| ----------------------- | ------------------ | ------- | ------- | ------------------------ |
| easy hump               | 0–10               | 49%     | **25%** | down-weight              |
| valley                  | 11–50              | 41%     | **35%** | up-weight                |
| hard-solved             | 51+                | 9%      | **15%** | up-weight (cap far tail) |
| hard / unsolved (hinge) | censored + anchors | 11.7%   | **25%** | up-weight + anchors      |


Headline: lift **hard/unsolved to ~1/4 of the signal**, keep the valley the largest block.

**Diversity / decorrelation (new — keep variety, not clustered look-alikes):** the labelled data
comes from solution *paths*, so consecutive on-path classes are ~1 S-move apart (near-duplicates).
Maximize **distinct** presentations per region: (a) **stride-subsample along each path** and **cap states kept per source path**, so we sample varied regions instead of dense chains; 

**Weighting mechanism:**

- `w_class = clip(target_band_share / actual_band_share, 0.25, 5.0)`, normalized to mean 1.
- Far-tail **d-o-t > 100 → extra ×0.5** (loose upper-bound labels, max 349).
- **No source reweighting by default**; expose a `--beam_weight 0.7` flag, apply only if val shows bias.
- Short-hard anchors (§3) get **×3**, capped at the global 5.0.

## 4. Censored loss bound B (tiered)

For unsolved rows we don't have a true d-o-t, only "greater than whatever the search reached." So
instead of a fake number we use a **one-sided hinge** that only punishes the model for predicting *too low*: `L = max(0, log1p(B) − pred)`. `B` is the floor we assert. Two tiers:

- `**B_soft = 48`** (= 90th percentile of solved d-o-t) for generic censored rows (might be solvable, just not found).
- `**B_hard = 150`** (= 99th percentile) for `tier ∈ {named, ak3_auto}` (believed non-trivializable, e.g. AK(3)).

Solved-d-o-t percentiles: p50=11, p75=20, p90=48, p95=81, p99=150, max=349. Each row stores its own `bound_B`. `B_soft` is the main knob → sweep {32, 48, 81} once the model trains. Full plain-language walk-through in CLARIFYING #4.

**Labels are upper bounds → the SOLVED loss must be asymmetric too.** Every d-o-t label is the *shortest path found* (min-aggregated), still only an upper bound — a row labelled 150 may have true d-o-t 50. So even for solved rows we must **not** punish *under*-prediction the way symmetric Huber would: use an **asymmetric loss** (under-prediction penalized lighter than over-prediction), so the model is free to predict the true-shorter distance and is corrected mainly when it over-shoots. Pair with ranking metrics (Spearman), which are robust to label looseness. When a shorter path later turns up, min-aggregation tightens the label automatically (PLAN §3). (This is the loose-label risk from CLARIFYING #5.)

## 5. Splits (class-level, seed=0, frozen — extends `dot_dataset.py`)

- `**hard_test`** — the hardest *solved* presentations: top 5% by d-o-t value, held out, never trained. Measures generalization to hard-but-solvable cases. **Caveat (CLARIFYING #5):** "top 5% by d-o-t" = top 5% by *known upper bound*, which may include loose labels (recorded 150, true 50). So on `hard_test` lead with **Spearman/ranking** (robust to looseness) and the **asymmetric loss** (§4) — don't penalize the model for correctly predicting *lower* than a loose label.
- `**counterexample_eval`** (NEW) — `tier=named` + the 8 cousins (+ 10% of ak3_auto when Phase 2 exists), frozen, never trained. The headline hardness-recognition metric. *Doing the discrimination well is a model-architecture problem, deferred to the model plan.*
- `**train / val / test`** = 80/10/10 of everything else (labelled + censored + remaining anchors). Censored go into train (hinge needs them) and val/test (to measure the hinge).
- **Probe assertion (build-time):** `(YXyxYx, x³y⁻⁴)` [solved, 12] → **train**; **AK(3)** → **counterexample_eval**. The look-alike pair must straddle train/eval.

Per-row schema (`dot_archive_v2.jsonl`): existing + `loss_type ∈ {regression,hinge}`, `bound_B`,
`weight`, `tier ∈ {solved,censored,named,ak3_auto,ak2_auto}`, `group`, `total_len`.
`dot_splits_v2.npz` adds aligned `weight`, `bound_B`, `loss_type`, fold arrays.

**#2 — "What are the 8 cousins?"**

- *Context:* §3b / §3c seed.
- *Answer:* The 8 **censored (unsolved) classes already in `dot_archive.jsonl` with total length ≤13**
— the short, AK(3)-neighbourhood hard cases. They are (canonical `r1 | r2`):
`YXXYx|YYYYXyX` (12), `YXXYx|YYYYYXXX` (13), `YXXYx|YYYXyxyX` (13), `YYxxyX|YYXyxxx` (13), `YXYXyX|YYXXXXX` (13), and three that carry **AK(3)'s exact first relator `YXYxyx`**:
`YXYxyx|YYYxxxx`, `YXYxyx|YXXYxxx`, `YXYxyx|YXyXYxx` (all 13).
*Are they AC-equivalent to AK(3)?* **Not proven — and I shouldn't have asserted "relatives" so firmly.** What is true: (1) they are total length ≤13, and (2) they were **not solved** by greedy (10M nodes) + beam. By the Havas–Ramsay classification the paper cites (every length-≤13 presentation is *either* trivializable *or* AC-equivalent to AK(3)), any length-≤13 presentation that is genuinely non-trivializable **must** be AC-equivalent to AK(3). So these 8 are **either** (a) actually solvable and the search just missed it, **or** (b) AC-equivalent to AK(3). The 10M-node budget makes (b) likely, and 3 of them literally share AK(3)'s first relator — but "unsolved within budget" is **not** a proof of non-trivializability, so it stays a strong suspicion, not a fact. Consequence: I treat the **named AK(n)** as the *gold* (proven-open) counterexample_eval, and tag these 8 as `group=short_hard` *suspected* hard (lower confidence) — useful anchors, but not asserted as AK(3)-equivalent.

**#3 — "What are the Length-14 ×2 and the cousins in the trap-set? If everything is canonicalized,
why list them separately?"**

- *Context:* §3d.
- *Answer:* **Nothing in the trap-set is "created/generated"** (except the Phase-2 automorphic
images, which are out of v1). The trap-set is just a **collected list of canonical keys** of
presentations we *already know are hard*, from three independent sources:
  1. `**AK(3) … AK(8)*`* — written straight from the formula `AK(n)=⟨x,y|xyx=yxy, xⁿ=yⁿ⁺¹⟩` (6 presentations, lengths 13,15,17,19,21,23).
  2. `**Length-14 #1/#2**` — **two completely separate presentations**, *not* AK(n) and *not* derived from AK(3). They sit in the lab's `counterexamples` dict in `greedy_search.ipynb`, and are merely *named* "Length 14" because their total length happens to be **14 letters**. The "14" is their own size — it has nothing to do with AK(3)'s 13. They are: `Length 14 #1` = raw `(XyyxYYY, XyxxyXX)`; `Length 14 #2` = raw `(XyyxYYY, XyxxYXX)`.
  3. **The 8 cousins** — short unsolved rows *already in `dot_archive`* (listed in #2).
  Why list them separately at all? Because canonicalization maps each presentation to **one** key, but **different presentations have different keys** — AK(3), the two Length-14, and the 8 cousins are all distinct classes, so canonicalizing does *not* fold them together. The trap-set is simply the union of those distinct keys (6 + 2 + 8 = **16**), and you can verify any one by running `canon.canon_key` on the raw pair given above. At search time we canonicalize a visited state and ask "is this key in the set?" — that's the whole mechanism.

**#4 — "Explain loss bound B / p90 / how it connects to our d-o-t data / what we do for AK(3)."**

- *Context:* §4.
- *Answer:* For **solved** presentations we have a real (upper-bound) d-o-t, so the model is trained to match it (regression). For **unsolved** ones (AK(3) etc.) we have **no number** — only "harder than anything the search reached." We must not invent a number (telling the model "AK(3) = 200" is a lie that hurts). Instead we use a **one-sided hinge**: we pick a floor `B` and only penalize the model if it predicts **below** `B`; predicting `B` or anything higher is free. So the model learns "this is *at least* this hard" and stays free to say "very hard / maybe never." `p90 = the 90th percentile of the d-o-t values of the SOLVED data` (= 48: 90% of solved cases have d-o-t ≤ 48). Using `B_soft = 48` says "this unsolved case is at least as hard as the top 10% of
everything we *did* solve." For the believed-non-trivializable ones (AK(3)-family) we use a higher floor `B_hard = 150` (= p99) — "deep in the hard regime." That's the whole connection: B is read off the solved data's percentiles, and it's a floor, not a target.
*How would it ever learn ~250 for AK(3)?* Three ways, none of which the hinge blocks (the hinge is a **floor**, not a cap — predicting *above* B is free):
  1. **Generalization from the labelled hard tail.** We *do* have solved presentations with real d-o-t up to 349. The DRT learns "this kind of structure costs ~~200, that kind ~300." If AK(3) is structurally close to solved cases that needed 250, the model **extrapolates** AK(3)→~~250 on its own. The 150 floor just stops it from going *below* 150; the actual large number comes from this regression signal. (This is exactly why §3's anchors + a populated hard tail matter — without high-d-o-t solved examples the model has nothing to extrapolate *to*.)
  2. **Censored→labelled conversion.** If any future search (beam/DAgger, PLAN §7) ever finds a path for AK(3) of length 250, that becomes a **real label** and the model learns it directly; min-aggregation then pins it (and tightens if a shorter path appears).
  3. **Moving the floor.** B is a hyperparameter — as we get better lower-bound evidence (e.g. exhaustive search to depth D with no solution ⇒ d-o-t > D), we can raise `B_hard`, sharpening the floor over time.
  So the model isn't "stuck at 150": 150 is the minimum we *assert*, and better predictions come from structural generalization + any future found paths, improving each DAgger round.

**#5 — "What is `hard_test` and the 5% d-o-t?"**

- *Context:* §5.
- *Answer:* `hard_test` is a **held-out test set of the hardest SOLVED presentations**: we sort all labelled classes by their d-o-t and take the **top 5% (largest d-o-t)** — the hardest-but-still-solvable cases. They are **frozen** (never used in training), so the model's error on them is anhonest measure of whether it generalizes to the hard hump (not just easy cases). It's separate from `counterexample_eval`, which holds the *unsolved* AK-family; `hard_test` = hardest solved,`counterexample_eval` = known unsolvable.
*You're right — this is a real risk, and it changed the plan.* Labels are **upper bounds** (shortest path *found*), so a row labelled 150 may have true d-o-t 50, and a model that correctly predicts 50 should **not** be punished. Three guards, now written into §4/§5:
  1. **Asymmetric loss (new in §4):** under-prediction is penalized *lighter* than over-prediction, so predicting the true-shorter distance is allowed; the model is corrected mainly when it over-shoots. (Symmetric Huber, which I had before, was wrong here — your point fixed it.)
  2. **Lead with ranking on `hard_test`:** report **Spearman / pairwise ordering** as the primary `hard_test` metric, not MAE — ordering is robust to a few loose labels even when the absolute number is off.
  3. **Self-tightening:** min-aggregation means the moment any shorter path is found, the label drops toward the truth (PLAN §3), so loose labels get *corrected over time* rather than baked in.
  Net: `hard_test` measures "does the model rank the genuinely-hard above the easy," not "does it reproduce a possibly-loose 150" — exactly to avoid the bias you describe.
