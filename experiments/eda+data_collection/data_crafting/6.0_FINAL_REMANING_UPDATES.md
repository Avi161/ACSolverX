## 2. The pivot — mine basins, don't censor by membership

**Why membership-censoring is the weak branch.** I read the lab greedy solver
(`greedy_search.ipynb::ACRelatorSolver`): it is a **best-first priority queue with priority = total
relator length**, and it **already canon-dedups** via `visited` keyed on `canonical_pair_nj`. So:

- A "basin" is **already a set of distinct canonical classes** (not rotation duplicates), **biased toward
  low length** — our A.2 "need to dedup" worry is *partly pre-solved*; the residual is *structural*
  correlation, handled by a per-basin cap.
- `visited` is **per `solve()` call** → there is **no cross-basin dedup** across the 550 roots, and
  AC-equivalent roots produce **overlapping** basins. The effective number of hard germs is small
  (≤ ~261 distinct classes `[unverified — Field Advisor]`), so **group-split by root class** and **cap
  per basin** at merge time.
- Censoring every basin node at a flat `d-o-t ≥ B` makes a **gradient-free plateau** — useless for the
  actual goal (*escaping* the basin), and the per-node `B` is **fabricated** (the search budget was spent
  from the root, not from that node).

**The mine instead.** Run beam/610 from a **stratified (length-band × basin) sample** of harvested nodes
(this is squarely within existing infra — `harvest_AC19`, `beam_harvest_pilot`, B=16,384 waves). One pass
does four things at once:

1. **Nodes that solve → real finite d-o-t labels for short-but-far states** — *exactly* the
   length-decorrelating signal the dataset lacks (today the regression head sees real targets only on the
   easy mass, so it learns *direction but not magnitude* on the hard tail).
2. **Nodes that don't → a principled per-node censor** at the budget actually spent ("unsolved by 610 +
   beam B=16,384, T=150" ≫ "appeared in a length-greedy frontier").
3. It **supplies the intra-basin gradient** a uniform censor cannot.
4. The **solve-rate *is* the false-positive measurement** of "basin = hard." Measure it; don't argue it.

**The deep AC reason this matters (not just hygiene).** The project's whole phenomenon is *graph-close to
trivial via a length-**increasing** detour.* A **length-priority** greedy is structurally **blind** to
those exits — so it will censor such a node **high** when its true d-o-t is **low**: the wrong direction on
precisely the states we care about. Mining both **fixes** and **measures** this. (This also **refutes** my
own earlier reassurance that "length-first greedy ⇒ few false positives" — that conflates *length*-distance
with *graph*-distance.)

**Off-action-space gotcha (must-fix).** Harvest with the **env's exact S-move and `L = 24`** — **not** the
greedy example's `max_len = 20`. Different length cap → different no-op boundary → different neighbor set;
mining at `max_len=20` produces distances that are **off the env's action space** and silently
inconsistent with everything else in the pipeline.

## 3. The load-bearing decision — MS held out vs. partitioned (decide FIRST)

**Sequencing rule:** *any presentation whose basin you harvest is burned for benchmarking.* So **freeze
the MS holdout before harvesting**, then harvest only from the complement.

**Where we stand today (good news):** our existing **4,954 censored are essentially MS-clean** —
canonicalizing both sets with `canon_key`, only **7 of 1190** MS classes appear in the merged d-o-t set
(all 7 unsolved there; verified provenance, memory `dataset-trivializability-provenance`). So **leakage is
a *new* risk introduced *only* by harvesting from the MS-unsolved roots** — it is not a problem with what
we already have. (One contamination footnote: the PPO-610 model was trained on the first **634** MS lines
of `AC19_extended.txt`, so MS path-length "ground truth" derived from it is **not** an independent oracle —
relevant to §4.)

| | **Option A — MS fully held out** *(recommended)* | **Option B — MS partitioned by class** |
| --- | --- | --- |
| Headline number | Preserves the literature-comparable **/1190** both papers report (companion PPO **431/1190**; GS **640/1190** per Lucas) | **Breaks /1190 comparability** |
| Statistical power | Lower (MS not in train) | Higher (uses the ~261 hard MS classes in training) |
| Hard-mass source | **AC-1M** (real labels, §7) + basins from **non-MS** roots | MS hard classes + basins |
| Leakage handling | **Dissolved** — no MS in train at all | Requires strict **class-level group hold-out + a *tested* leakage filter** on the harvest |
| Effort / risk | Low; clean story | Higher; one missed canon-key = silent contamination |

**Recommendation: Option A**, unless we have a specific reason to want the hard MS classes *in training*.
A keeps the publishable `/1190` number intact and removes the leakage tension entirely by sourcing hard
mass from AC-1M (§7) + non-MS basins. **This is the decision I need from you / the mentor.**

## 4. Two distinct benchmarks — do not conflate

The established protocol from the two papers, mapped onto our goal:

- **Recognition (hard vs. easy) = a *classification* problem.** Report **F1 / AUC**, benchmarked against
  the companion paper's **XGBoost-on-(neighborhood + persistence-barcode) F1 ≈ 0.96** and the
  **length-only floor ≈ 0.885** `[unverified — companion paper, via Field Advisor; our CLAUDE.md
  independently records XGBoost F1≈0.96]`. **That is the number to beat — not d-o-t regression.** (Caveat:
  the 0.96 was *binary GS-solvability* classification, a different task/label — it's a recognition
  baseline, it does **not** transfer to regression.)
- **Regression (d-o-t magnitude) = ranking on the *solved* subset only.** Use **Spearman / pairwise**,
  **never MAE** — the targets are **upper bounds** (the hybrid-beam shortening 13.77 → 12.39 `[unverified
  — Field Advisor]`, and your own 75%-PPO-shorter result, *prove* they move). This is the same
  reframing already baked into our `counterexample_eval` (5.1 §5).
- **The decisive test (the actual goal): solves-at-matched-*compute*-budget** when ranking moves by
  predicted d-o-t vs. by length. A win here is the real claim; a regression MAE is not. Match **compute,
  not params**; **≥ 3–5 seeds**, report variance.

## 5. Censored-set fidelity audit (Lucas's point 2, reframed)

Run greedy to its plateau on **(a)** the 550 MS-unsolved **and (b)** our existing **4,954 censored**, and
record which ones crack. This measures the **mislabel rate of the censored set** (how many "hard" rows are
actually solvable-but-missed), which directly calibrates how much to trust the hinge labels and where to
set `B`. It is *Lucas's point 2 = our point C*, just reframed from "what budget?" to "**how wrong are the
labels?**" Cheap; **do it first.**

## 6. Tie to our own length-confound finding (our specific upgrade)

Phases 7–9 established that **length is *the* confound** and that `solved_median_dot(L)` **rises
monotonically** (so AK(6/7/8)@19/21/23 are the *most* length-gameable — 5.1 §5). Therefore the value of
basin-harvest for *our* problem depends entirely on the **length distribution of the harvested nodes**:

- If the basins skew **long**, they **reinforce** the exact "long ⇒ hard" shortcut we are fighting.
- **So: measure basin-node lengths, then prioritize / stratify the *short* hard ones** (low total length,
  high *mined* d-o-t). Short-but-far is the gold the dataset lacks and the only thing that teaches the
  model **structure over length.**

This is the non-obvious upgrade to Lucas's generic "save a ton of these" — *which* ones we save is what
makes or breaks the length-decorrelation goal.

## 8. Architecture bake-off (for the model phase, not now)

Lucas's order is good and we keep it: **1D-CNN (circular conv + attention pooling) → plain transformer
(control) → dual-ring transformer** (the lab's established SOTA — "we never beat it"). Add:

- **Baselines that must be beaten before any net is interesting:** the **length-only floor** (~0.885),
  **XGBoost-on-features** (~0.96 recognition), and — critically — the **610 value head used *directly* as
  the search heuristic** (the redundancy baseline: if a fancy regressor only ties the value head we
  already have, it adds nothing).
- **Match compute, not parameter count;** **≥ 3–5 seeds;** report variance, not a single run.
- Keep recognition (classification) and regression (ranking-on-solved) as **separate** scoreboards (§4).
