# [2026-07-09] `MAX_RELATOR_LENGTH` cap-monotonicity: the tempting theorem has a vacuous guard [TRAP]
I derived: "raising the cap cannot change `solved`/`nodes_explored`, provided no popped state exceeded the smaller cap", intending to test it. The guard is uncheckable from the reported stats: `max_relator_length_expanded` is a **total over both relators** while the cap is **per relator**. Measured on ms640: idx 621 → `max_expanded_total = 25`, idx 630 → `31`, both against `cap = 24`. So the guard never fires on precisely the cases worth checking, and a test written around it would silently prove nothing. **Fix:** pin specific `(pres, budget, cap1, cap2)` pairs *measured* as identical. Originally 621 → 15,371 nodes / 500 moves and 630 → 12,429 / 79; both need budgets above the 10k ceiling, so the suite now pins **ms640 626 @ 8,000 → 6,517 nodes / 56 moves, identical at cap 24 and cap 48**, plus an unsolved rep at 3,000. (The 15,371 / 12,429 figures remain true, just untested — they cost 11 minutes.) **Rule:** before building a test on a derived invariant, check that its precondition actually holds on the fixtures — a guard that is always false makes a green test meaningless.
