# [2026-07-09] Test budgets are capped at 1,000 nodes — and that costs nothing [WORKS]
The first `--runslow` took **16 minutes**, essentially all of it one line: the golden sweep ran ms640 620-639 at **30,000** nodes, and the ~10 that never solve burn the full budget at ~514 nodes/s on those long relators (~58 s each). `test_golden` then re-ran the same sweep.
**Why a small budget loses nothing.** The heap key `(total, depth, key)` is a *strict total order* — keys are unique per state and the closed set admits each state once — so pop order cannot depend on push order **or on the budget**. A search at budget `B` is exactly the first `B` pops of a search at any larger budget. Therefore a presentation that solves within `B` returns a bit-identical stats dict at every larger budget. Verified, not assumed: `greedy_search` at 700 vs 1,000 on ms640 466 returns `==` dicts (614 nodes, 101-move path), and ms640 0-7 at 200 vs 1,000 likewise, on **both** solvers. Below the solve point the complementary fact holds: `nodes_explored == budget` exactly, and `max_relator_length*` grow monotonically with the budget.
**Result:** `MAX_BUDGET = 1_000`, enforced in `regen_golden.py` and asserted by `test_golden.py::test_no_golden_entry_exceeds_the_budget_ceiling`. **554 of 640 ms640 presentations solve inside it**; the deepest costs 990 nodes (idx 551, 69 moves) and the longest path is 101 moves (idx 466, 614 nodes) — both pinned at cap 24 and cap 48. Golden regeneration: 705 s → **21 s**. Full tier: ~16 min → **~2 min**. The generalization is now an explicit *test* rather than an implicit assumption.
**One exemption, handled without raising a budget:** the progress callback fires every `_HB_CHECK_EVERY = 1024` pops, so a 1,000-node search is silent. Rather than bump the budget, the tests monkeypatch `greedy_baseline._HB_CHECK_EVERY` to 100 (both solvers read it as a module global on every pop) and a separate test pins the real constant at 1024 and asserts the silence.
**Rule:** when a suite is slow, look for the property that makes the expensive case redundant before trimming coverage. Here "a prefix of a longer search" turned a runtime cut into an extra theorem. If you want a deeper anchor, find a presentation that solves in *fewer* nodes — never raise the budget.
