## Lessons learned

### [2026-07-08] Colab Drive path — mount root is not writable [TRAP]
`os.makedirs("/content/drive/acsolverx_results")` → `OSError [Errno 95] Operation not supported`. The Drive FUSE mount root `/content/drive/` rejects directory creation. **Rule:** all Drive output paths must be under `/content/drive/MyDrive/...` (or a Shared drive), never `/content/drive/` directly.

### [2026-07-08] Colab SETUP cloned the repo nested (ACSolverX/ACSolverX/…) [TRAP]
Cause: SETUP did `os.chdir(REPO_ROOT)` into the repo, then a *re-run* checked `os.path.isdir(REPO_DIR)` with a **relative** `REPO_DIR` from inside the repo → clones again one level deeper each run. **Rule:** anchor to an absolute base first (`os.chdir("/content")`) and derive `REPO_ROOT = os.path.join("/content", REPO_DIR)`; make clone/pull idempotent. For local, walk *up* to the repo root (dir holding `experiments/` + `data/`), don't assume cwd.

### [2026-07-08] Pushing notebook changes does NOT update the user's running Colab cells [TRAP]
The notebook cells executing in Colab are the user's editor buffer, independent of the repo copy. Only `.py` files re-pulled by SETUP's `git reset --hard` auto-update; `cfg` comes from the user's CONFIG cell. **Rule:** when a fix touches a notebook cell, tell the user to re-open the notebook from GitHub (or hand-edit the cell) — a push alone won't reach them. Prefer putting logic in importable `.py` modules so fixes propagate via git.
**[2026-07-09 CORRECTION]** "`.py` files re-pulled by SETUP auto-update" is **FALSE** and this claim directly caused a wasted debug cycle — see the `git pull is not a reload` lesson below. A pull rewrites the file on disk; the already-imported module object survives in `sys.modules`. Putting logic in `.py` modules propagates the *file*, not the *loaded code*.

### [2026-07-09] `git pull` is NOT a module reload — stale sys.modules ate a shipped fix [TRAP]
User pulled a heartbeat fix, re-ran SETUP + RUN, and still saw the old (broken) behaviour. Cause: SETUP's `git reset --hard` updated `experiments/run_baseline.py` on disk, but `experiments.run_baseline` was already in `sys.modules` from the previous RUN, so the RUN cell's `from experiments.run_baseline import run_dataset` returned the **stale** module. Proven locally: import, rewrite file, re-import → still the old value; after `del sys.modules[...]` + `importlib.invalidate_caches()` → new value. Interrupting a cell does NOT clear `sys.modules`; only a runtime restart does. Worse, the diagnostic line (`heartbeat: nodes/s every 90s`) existed in BOTH commits, so the output looked like the new code was running. **Fix:** SETUP now purges `experiments*` from `sys.modules` + `importlib.invalidate_caches()` right after `os.chdir(REPO_ROOT)`. **Rules:** after pushing a `.py` fix, tell the user to **Runtime → Restart session** (or re-run the patched SETUP), never just "re-run the cell"; when diagnosing "my fix didn't take", first prove WHICH code is loaded (`inspect.signature`/`inspect.getsource` of a symbol that exists only in the new commit) rather than trusting a log line that both versions emit. Beware `.pyc` staleness too: a rewrite with identical size within the same mtime second reuses the cached bytecode (hit while writing the repro).

### [2026-07-08] HIGH_SPEEDUP heavy solver: boxing was the cost, memory was the cap [WORKS]
Heavy runs (261 unsolved reps, 1M nodes, mrl=48) were infeasible: measured **43.4× discovery ratio** (30k pops → 1,302,816 states), **554 B/state** → ~43M states / **~25 GB per presentation**, ~14 min each ⇒ ~60 h serial. `cProfile` showed the time was **numba boxing**, not math: 3.5M `_numba_unpickle` calls per 8k pops, because `get_neighbors_with_moves_nj` returns a Python list of ~440 tuples-of-ndarrays per node. And 50 GB fits exactly ONE presentation, so cores sat idle — **memory reduction is what unlocks parallelism, not a separate fix.**
**Solution** (`HIGH_SPEEDUP` config bool, default False; normal path bit-for-bit unchanged): (1) drop `new_seen` — it only fed the final `min()`/`max()` scan → track min/max incrementally; (2) `visited` becomes a `set`, `move_in` gone (no path) → solved presentations are re-solved by the normal solver **after the pool tears down** (it needs full RAM) and get `path_recovered: true`; (3) packed `bytes` keys; (4) `expand_node_nj` does neighbours+reduce+canonicalise inside numba returning one fixed-width `uint8` buffer (use `np.empty`, not `np.zeros` — only written cells are read), plus `multiprocessing` over presentations with the **parent as sole jsonl writer** (no lock, resume untouched).
**The key insight that made (3) free:** the packed key must SORT IDENTICALLY to `(str, str)`, because the heap tie-break is `(priority, depth, key)` and `depth` is fixed by first-visit order. A 1-byte order-preserving alphabet (`X→1 < Y→2 < x→3 < y→4`, matching ASCII) with a `0x00` separator (below every symbol, reproducing str's shorter-prefix-is-smaller) does exactly that — verified on 3,681 pairs. Also: `expand_node_nj` must enumerate in the same `target→jsign→k1→k2` order, else `depth` shifts and pop order changes.
**Verified:** 400 runs (200 pres × 2 cyclic modes) — 0 mismatches on `solved`/`nodes_explored`/`min_relator_length`/`max_relator_length`/`max_relator_length_expanded`; `min_relator` and `max_relator_expanded` strings identical too. Only `max_relator` differs (353 cases) — all **equal-total-length ties**, because `max(set, key=…)` picks arbitrarily among ties while the incremental scan picks the first. Parallel == serial; recovered paths replay to trivial. **Measured: 554→213 B/state (2.6×), 678→281 µs/node (2.41×), identical discovered counts (1,302,816).** ⇒ ~9.2 GB & ~4.7 min per presentation @1M ⇒ 4 workers on 50 GB ⇒ ~5 h instead of ~60 h.
**Rules:** `HIGH_SPEEDUP` must NOT enter `_run_prefix` (it doesn't change results — like the `use_*` toggles). Heavy mode is *wasteful on easy data* (every solve triggers a re-solve) — use it only for hard sweeps. It does **not** make the 261 solvable; it only completes the experiment.

### [2026-07-08] Date in jsonl filename broke cross-day resume [TRAP]
`_resolve_paths` stamped `..._{mm_dd_yy}.jsonl` (per the plan spec) AND used that exact name as the resume key. A run continued the next day (or after a Colab disconnect spanning midnight) computed a NEW date → new filename → `RESUME` found 0 prior rows → "0 already done, 640 to run", restarting from scratch (user hit this: run named `..._07_09_26` re-ran everything). The whole point of the jsonl is disconnect resilience, which the date defeated. **Fix:** split the stem into a date-less `_run_prefix` (the full search identity) + the date suffix. On `RESUME`, glob `out_dir` for `{prefix}*.jsonl` (excluding `_paths.jsonl`) and reattach to the existing file with the MOST rows, regardless of its date; only create a fresh dated file when none exists. Verified: a seeded old-dated file (4 rows) is picked up by a run on a later date → "4 already done, 6 to run", continues into the same file, no new file. **Rule:** never put a wall-clock date (or any per-invocation-varying token) in a filename that also serves as a resume key; resume must key on identity only.

### [2026-07-08] W&B "steps must be monotonically increasing" on pipeline re-run [TRAP]
Re-running the pipeline logged `run.log({"solve_rate":...}, step=pres_id)`. With a deterministic `run_id` + `resume="allow"`, a re-run RECONNECTS to the earlier run whose global step is already ≥640 (the finish-time Table/panel logs, which have no `step=`, bumped it past the last pres_id). Logging `step=419 < 640` → every row rejected with `Tried to log to step 419 that is less than the current step 640`. **Fix:** stop pinning solve_rate to the global step. After `wandb.init`, `run.define_metric("pres_id")` + `run.define_metric("solve_rate", step_metric="pres_id")`, and log `run.log({"pres_id": pid, "solve_rate": rate})` with NO `step=`. Now the global step auto-increments monotonically (never rejected) while the chart uses pres_id as x-axis. **Rule:** never pass a domain index (pres_id) as W&B `step=` on a resumable run; define a custom `step_metric` instead. NOTE: offline mode (wandb 0.28) does not emit this warning, so it can't be reproduced locally — this is the W&B-documented fix (wandb.me/define-metric).

### [2026-07-08] Hoist k1-independent rotation out of neighbour-gen inner loop [WORKS]
`get_neighbors_with_moves_nj` recomputed `np.roll(oj, 2*k2)` inside the `k1` loop, but the other-relator rotations don't depend on `k1` — O(len_i*len_o) rolls → O(L³). Precomputing `rots_o = [np.roll(oj, 2*k2) for k2 in range(len_o)]` once before the `k1` loop drops it to O(L²). Added `cache=True` to the plain `@njit` entry points (persists compiled code, ~3s-faster fresh-process startup). Verified behaviour-identical: 400-presentation fingerprint (solved/nodes_explored/path_length/path_moves) 0 mismatches vs pre-change baseline; neighbour set still == Def-2.1 oracle; full 640×2 replay still 606/580 solved, 100% lossless. Speed: neighbour-gen 92.9→66.2 µs/call (**1.40×**) at len-12 relators, more at longer L. **Rule:** in nested rotation loops, hoist any roll/transform that's independent of the outer index; whole-solve speedup < neighbour-gen speedup because reduce/canonical/string-keying per node are untouched (finding #3 = packed-bytes keys, still open).

### [2026-07-08] reduce_relator_nj empty-word OOB + str_to_arr('') malformed [TRAP]
Independent-oracle verification (Opus subagent, 400k reduction checks + full-640×2 replay) found `reduce_relator_nj` returned a garbage length-1 relator (`'Y'`) whenever a word free-reduced to the **empty word**. Root cause: the prototype's boundary branch `rel_list[0] = rel[add_index + 1]` does an **out-of-bounds read** when the cancelling partner is the last symbol (numba has no bounds-checking → garbage), and the algorithm had no representation for length-0. Latent on ms640 (never triggers on the 1186 solved paths, `oracle_diverged=0`) but a genuine defect: a substitution whose reduced relator is the identity (relators equal up to cyclic/inversion) hits it. **Fix:** rewrote as a plain stack free-reduction (returns `(0,2)` empty on full cancellation); kept the original cyclic tail (guarded `length>1`). Also fixed `str_to_arr('')` which returned a malformed 1-D `(0,)` array (crashes numba on the (n,2) signature) → now returns well-formed `(0,2)`. With both, empty relators are safe dead-ends (neighbour-gen yields 0 from a len-0 relator; never trivial). **Re-verified:** 80k oracle-fuzz 0 fails; full 640×2 replay still 606/580 solved, 100% lossless (unchanged). **Rule:** stack-based reduce is the safe form; any `rel[i+1]` peek in numba needs an explicit `i+1 < n` guard; `str_to_arr` must always return 2-D even for ''.

### [2026-07-08] Store solved paths as Definition 2.1 moves, not string pairs [WORKS]
Storing each path step as two full relator strings is bulky and NOT self-documenting (you can't read off which substitution was applied, because every state is Booth-canonicalised — rotation/inversion/possible r1↔r2 swap). Paper Definition 2.1 (`two_hump_paper.txt:229`): a substitution move is `(i,j,k1,k2) ∈ {1,2}×{−1,1}×Z×Z`, replacing `r_i` by `rot_{k1}(r_i)·rot_{k2}(r_{3−i}^j)`. Our `get_neighbors_nj` already computes exactly these params (candidate idx→j, roll amounts→k1,k2, which append→target) — it just discarded them. Added `get_neighbors_with_moves_nj` + `replay_move_nj`; threaded `move_in` dict through `solve()`; `greedy_search` now returns `path_moves` as compact `'target_jsign_k1_k2'` strings (~4 ints/step). **Key correctness fact:** the move is relative to the *canonical parent*, so decoding = REPLAY (`moves_to_states`: apply raw move → reduce → canonicalise, per step), NOT a diff of the stored strings. Verified lossless: replay reproduces the string-path for all 80 solved presentations across both cyclic settings. Paths file stores `{pres_id, r1, r2, path_moves}` (r1/r2 needed as the replay start). Config `PATH_FORMAT` = "moves" (default) | "strings" | "both".

**Parameterization must match Definition 2.1 (invert the OTHER relator).** The notebook `get_neighbors_nj` always builds `rot(r1)·rot(r2^j)` and drops it into either slot — so for `target=2` it inverted r2 (the target) instead of r1, contradicting Def 2.1 (`r_i ← rot(r_i)·rot(r_{3−i}^j)`: the *other* relator carries j). This is only a LABELLING difference (the canonical neighbour SET is provably identical — 0/30 presentations differ vs a from-scratch Def-2.1 generator), because canonicalisation quotients out cyclic rotation AND whole-relator inversion. But for paper-faithful, human-readable tuples, `get_neighbors_with_moves_nj`/`replay_move_nj` were rewritten to the Def-2.1 form (target `i` → invert `r_{3−i}`). Re-verified after: neighbour sets still 0/30 vs notebook gen, replay still 80/80. `2_-1_0_0` now reads correctly as `r2 ← r2·r1⁻¹`. **Rule:** when storing moves against a paper's definition, test the stored tuple decodes to the definition's exact operation, not just that replay round-trips — round-trip only proves internal consistency, not agreement with the literature.

### [2026-07-09] Forked workers hang in numba's compiler after wandb.init() — warm the JIT in the parent [TRAP]
Instrumentation finally localised the "silent heartbeat": `[hb] pres N started` printed for all 4 workers (so the pool, the `mp.Queue` and the main-thread drain were all fine), but `msgs_received` froze at 4 and `live=0` **forever** — no worker ever reached its first 1024-node tick, which in debug mode (`first_after=0`) emits immediately. The `start` message is enqueued *before* `greedy_search`, so the children were alive and then blocked at their first numba call. Cause: `wandb.init()` makes the parent multi-threaded, `mp.Pool` forks it, and a child forked from a threaded parent can deadlock inside numba's compiler on a lock held at fork. Cold compile is only **2.5 s** single-process, so "4× slow JIT" never explained 65 s+ of silence. Every local test passed because none of them ran wandb. **Fix:** compile the kernels in the PARENT before forking (`greedy_search(r1, r2, node_budget=2, high_speedup=True)`), so forked children inherit compiled code and never enter the compiler; it also turns 4 cold compiles into 1. Added `MP_START_METHOD` (None = OS default; `"forkserver"`/`"spawn"` fork from a clean single-threaded process) as an escape hatch. Verified with a threaded parent + wiped numba cache across fork / forkserver / spawn: warm in 2.8 s, first sample at t≈0.5 s. **Rules:** never fork a pool out of a process that a library (wandb, ipykernel) has made multi-threaded without warming every lazily-compiled/locking resource first; when a bug only appears in the user's environment, reproduce the environment's *distinguishing* property (here: threads before fork) rather than re-testing the clean case. Tagged queue messages (`start`/`sample`/`done`) are what made "worker never ran" separable from "worker alive but stuck" — instrument the boundary, don't guess across it.

### [2026-07-09] Heartbeat silent for 2x the interval — tested at 2s, shipped at 90s [TRAP]
User set `HEARTBEAT_EVERY_S=90` on the real 261x1M run and saw NOTHING after 90s. Two compounding phase bugs, both invisible at the `every=2` I tested with: (1) `_Heartbeat` waited a **full** `every` before its FIRST sample, and the worker's clock starts before numba JIT eats the first seconds; (2) `_hb_monitor` initialised `last_print = time.time()` and then ran `last_print = now` **even when it printed nothing** — so the T+90s tick found `live` empty (the worker's first sample lands moments later), burned the whole period, and the first block appeared at T+180s. Whether it was 90s or 180s was a pure race on which arrived first. **Fixes:** `last_print = 0.0` (first arriving sample prints at once); `continue` WITHOUT advancing `last_print` when `live` is empty; `_Heartbeat` emits its first sample after `min(first_after=10s, interval)`; and pool workers sample at `hb_s/2` (the parent prints every `hb_s`, so sampling at `hb_s` would build `agg` from rates up to a full period stale). Verified with a timestamped wrapper: first block at 8.5–11s, then exact cadence, across serial / fork / spawn. **Rules:** a rate-limiter's *first* emission is a separate design decision from its steady-state period — long silence at startup is indistinguishable from a hang. Never advance a "last fired" timestamp on a tick that produced no output. And **test periodic code at the production interval** (or with a timestamped log): a 2s test hides every phase/race bug that a 90s interval exposes.

### [2026-07-09] Live nodes/s heartbeat — a pool worker cannot print into a notebook [TRAP]
`PROGRESS_EVERY` only prints when a presentation *finishes*; at a 1M budget that is minutes of silence, so there was no way to see CPU throughput mid-solve. Added `HEARTBEAT_EVERY_S` (0 = off): both solvers take an optional `progress` callback, invoked every `_HB_CHECK_EVERY = 1024` pops; the callback rate-limits itself by wall clock. Overhead unmeasurable (146 vs 152 µs/node, heartbeat-on measured *faster* — pure noise). **The trap:** in the parallel (`HIGH_SPEEDUP` + `n_workers > 1`) path a worker's `print()` is silently DROPPED in Jupyter/Colab — ipykernel's stdout proxy forwards through a background `pub_thread` that a forked child does not inherit. Workers therefore `put_nowait` samples onto an `mp.Queue` (passed via `Pool(initializer=...)`, which survives both `fork` and `spawn`) and a daemon thread in the PARENT drains and prints. **Second trap:** dropping a finished presentation by a `2*every` staleness window is wrong — its last rate keeps inflating the `agg nodes/s` and the "N solving" count for two more heartbeats (observed live). `_solve_one` now sends a `(pres_id, None)` done-sentinel in a `finally`; the staleness window survives only as a backstop for a worker that dies. **Rules:** verify the parallel path by FORCING `N_WORKERS=2` (on 16 GB `_auto_workers` returns 1, so `n_workers > 1` never fires and the whole Queue/monitor path goes untested); test under both start methods (macOS defaults to `spawn`, Colab to `fork`); `HEARTBEAT_EVERY_S` is result-neutral so it must stay OUT of `_run_prefix`. Verified `progress=None` vs a live callback returns bit-identical stats dicts on 4 presentations × both solvers.
